The NodeJS Philosophy

Node.js has become a popular platform for application development. The community is growing and there are certain principles
that define how applications are built and designed for nodejs.

These philosophies can be really helpful when you are working in your node.js app.

None of the following rules are imposed, and should be applied with some common sense.

Small Core

The foundation of node.js is to have a very small set of core functionalities.

This principle has allowed the developer community to build broader set of features, iterate faster and community is not bound to any slow-evolving solution.
Because of this principle, we have so many amazing npm libraries.

Also, since we have bare minimum core functionalities, the maintainability is much easier.

Small Modules

Node.js promotes the concept of modules which is the smallest unit of reusable code. It promotes designing small modules in terms of size of code and scope of the code.

This is the exact same principle that we see in Unix:

  • Small is Beautiful
  • Make each program do one thing well

Some of the developers might think that this approach is counter-productive. But, with the rapid changes in technology it provides us with:

  • Better code reusability
  • Easier to understand and use
  • Simpler to test and maintain
  • Perfect to share with Browsers

Its not uncommon to find several npm modules that perform just one function and is less than 100 lines of code!

Small Surface Area

OK, so the modules have to be small in size and scope. What the hell is - small surface area ?

Well, the intent is to - expose bare minimum functionality. The idea is to use the functionality and not to extend the functionality.
Yes, you read it right : lock down the internals of the module and expose minimal features.

It might sound inflexible but it offers :

  • simplified implementation
  • API is less prone to erraneous usage
  • increasing usability
  • better maintainability

Simplicity

You must have heard of the KISS - Keep It Simple, Stupid principle.

Simplicity is the ultimate sophistication.

Leonardo da Vinci

Designing a simple software vs a full-featured software has its own benefits.
Pure Object Oriented Designs try to mimic the real world without complexities and imperfections of the real world.
Its much better to get something working sooner with reasonable complexity, instead of trying to create a near-perfect solution with huge effort and tons of code to maintain.

I prefer uncomplicated practical approach than a pure flawless design involving huge effort.
